# Design-Knowledge Artifact Schema

This is the **strict template** for the file the `init` mode produces. It is
written to the user's repo at `.design/knowledge.md` and is the single source of
truth that the `review` and `pass` modes consume.

The artifact has one job: **be sharper than the base model's generic design
sense.** A critic fed thin knowledge produces design-blog advice ("improve
contrast, use whitespace") that gets ignored. Every section below exists to
force project-specific detail. If a section would only contain generic truisms,
the extraction failed — go back to the inputs and dig.

---

## Rules for filling the template

1. **Be specific to THIS project.** Name real components, real token values, real
   files. "Use the spacing scale" is useless; "Use `space-2` (8px) / `space-4`
   (16px); never raw px" is usable.
2. **The highest-value field for any component is `Use instead of`** — what the
   reviewer should flag when someone hand-rolls or mis-picks. Always fill it.
3. **Tag every inferred claim** with `⚠️ INFERRED — confirm`. Anything you
   derived rather than read directly from an input gets the tag, so a human can
   ratify it in one pass. (`update` may also add `⚠️ STALE — confirm removal` on
   an entry whose source has disappeared — flagged for a human to remove, never
   silently deleted.)
4. **Cite the source per entry, summarize it in the header.** Each entry already
   carries its exact origin (a component's `Import:`, an anti-pattern's file path,
   a token's defining file) — that's what a reviewer uses to trace a finding. So
   the header `Sources:` line should be a **scannable summary, not an exhaustive
   file list**: use globs and counts (e.g. `components/ui/* (28 components)`,
   `app/global.css (tokens)`), not 40 comma-separated paths. If an exhaustive list
   is genuinely useful, put it in a collapsed `<details>` at the end — never inline
   in the header.
5. **Omit empty sections honestly.** If the user fed no brand doc, write
   `Principles: none provided` rather than inventing a stance. Never pad.
6. Keep it scannable. The critic reads this on every run — terse beats verbose.

---

## The template

````markdown
# Design Knowledge — <Project Name>

> Generated by turtleneck (init mode).
> Sources: <a scannable summary — globs + counts, e.g. `components/ui/* (28
>   components)`, `app/global.css` (tokens), `app/layout*.tsx`. NOT an exhaustive
>   file list; per-entry `Import:`/file refs carry the exact origin.>
> Last updated: <date>
> Review the `⚠️ INFERRED — confirm` tags below and delete the tag once confirmed.

## Components

For each notable component, one block. Order by how often it's mis-used.

### <ComponentName>
- **For:** <the one situation this is the right choice for>
- **Use instead of:** <what people hand-roll or wrongly reach for — the thing
  the reviewer should flag. e.g. "a raw `<button>`", "a div with onClick", "the
  old `LegacyModal`">
- **Import:** `<exact import path or registry name>`
- **Key props / variants:** <only the ones that carry design meaning, e.g.
  `variant="primary|secondary|ghost"` — not an exhaustive prop dump>
- **Notes:** <gotchas, when NOT to use it>

<repeat per component>

## Tokens / Scales

The allowed values and how to reference them. Flag off-scale usage in review.

### Color
| Token | Value | Use for |
|-------|-------|---------|
| `<--color-primary>` | `<#hex>` | <primary actions> |
| ... | ... | ... |
- **Reference as:** <CSS var / theme path / Tailwind class — the canonical way>
- **Never:** <raw hex in components / off-palette colors>

### Spacing
- **Scale:** `<space-1=4px, space-2=8px, ...>`  (the full allowed set)
- **Reference as:** `<how>`
- **Never:** <arbitrary px / margins outside the scale>

### Typography
- **Type scale:** `<text-sm / text-base / text-lg ... with sizes>`
- **Weights allowed:** `<e.g. 400, 600 only>`
- **Never:** <off-scale sizes, disallowed weights>

### Radius / Shadow / other
- <only if the project has a meaningful, finite set>

## Principles

The project's actual design stance, distilled from brand docs / blessed examples.
2–6 sharp, falsifiable statements. Each must be specific enough that a reviewer
could point at a screen and say "this violates it."

- **<Principle name>:** <statement>. <how it shows up concretely>.
  - e.g. **Restraint:** Never more than two type weights on one screen. Whitespace
    over dividers — separate sections with space, not borders.
- <repeat>

If no brand/style input was provided: `Principles: none provided — relying on
tokens + components only.`

## Reference Examples

2–3 pointers to canonical "this is what good looks like" screens/components.
The reviewer uses these as the bar.

- `<path/to/screen-or-component>` — <why it's exemplary: what it does right>
- `.design/examples/<image>.png` — <what to notice in it>  (human-added)

## Anti-patterns (optional)

Concrete things this project has explicitly decided against. Strong BLOCKER fuel.

- <e.g. "No icon-only buttons without aria-label", "No nested cards">
````

## Worked example (abbreviated)

This is what a *good* filled section looks like — note the specificity:

````markdown
### Button
- **For:** any clickable action that triggers behavior (not navigation).
- **Use instead of:** a raw `<button>`, or a `<div onClick>`. Both are BLOCKERs.
- **Import:** `@/components/ui/button`
- **Key props / variants:** `variant="primary|secondary|ghost|destructive"`,
  `size="sm|md|lg"`. Default `primary`/`md`.
- **Notes:** For links styled as buttons use `<Button asChild><a/></Button>`,
  not a styled anchor.
````

Contrast with the **failure mode** the artifact must avoid:

````markdown
### Button
- **For:** buttons.
- **Use instead of:** other buttons.        ← useless; generic; delete and redo
````

If a generated block reads like the second one, the extraction was too thin to
ship. The reviewer built on it will produce ignorable advice.
